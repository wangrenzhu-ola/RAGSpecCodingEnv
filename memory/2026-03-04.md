# Daily Memory: 2026-03-04

### 评估hkt-memory与RAG和GraphRAG差距 (13:44:31)
- 当前实现属于经典RAG范式：SQLite FTS5+向量检索、Weighted Hybrid、MMR、时间衰减。
- 代码证据在 hkt-memory/scripts/vector_store.py：hybrid_search采用加权融合并支持mmr与decay。
- 与GraphRAG主要差距：缺少实体关系抽取、知识图谱构建、图遍历检索、社区摘要与全局/局部查询模式。
- 结论：符合RAG定义，距离GraphRAG约一整层数据结构与索引管线能力。

### OpenClaw为何未采用GraphRAG (15:33:48)
- OpenClaw记忆系统当前定位是Markdown为真源+SQLite索引的务实方案，强调简单、快速、可降级。
- docs/concepts/memory.md 明确当前是向量+BM25加权融合，并注明不是IR理论最优而是工程上简单快速。
- 源码src/memory/hybrid.ts 仅实现向量与关键词融合、时间衰减、MMR，不含图谱构建与图遍历。
- 结论：不是否定GraphRAG，而是当前阶段优先成本/延迟/可维护性，图能力可作为后续增强。

### RAG与GraphRAG哪个更像人类记忆 (15:39:20)
- 结论：两者各像一部分，人类记忆更接近混合体。
- 当前OpenClaw式混合检索更像人类的快速联想与近期偏置（语义相似+关键词+时间衰减+去重）。
- GraphRAG更像人类的语义网络与关系推理，适合多跳关联与结构化解释。
- 最佳工程方向是默认混合RAG，复杂问题触发图扩展的分层策略。
